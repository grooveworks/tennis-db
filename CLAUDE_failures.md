## v2 → v3 → v4 で繰り返した失敗 (再発防止のため記録)

### F1. 対処療法の連鎖 (CLAUDE.md R1 違反)
問題発覚 → パッチ → 新たな歪み → パッチ、の無限ループ。  
結果 v3 は Sessions タブの細部だけ発達し、Gear / Plan / Insights は Placeholder のまま。

### F2. 判断をユーザーに投げる (R2 違反)
「A/B/C どれ?」を連発して素人に技術判断を求める。指摘されて「判断します」と誓った後、また投げる。

### F3. 事前調査不足で動かない DoD (R3 違反)
環境制約 (preview/file://) を調べず、Firebase 動作を DoD に入れる。PowerShell の `-join` / `Set-Content` の罠で時間浪費。

### F4. 謝罪を軽く流す
「ミスでした、でも○○なら OK です」と言い訳にすり替える。謝罪と次アクションを同じ応答で混ぜる。  
**対策**: 謝罪は単独段落、その後に改めて次の行動を提示。

### F5. 専門用語で素人を置き去りにする
「Stage」「Domain」「カバレッジ」等を平易化せずに使う。ユーザーが理解していないのに承認を求める。

### F6. デザインを軽視する (R4 違反)
機能優先で作り、デザインは後回し。v3 では UI 統一感なく終わった。

### F7. 文書を書いて満足する (これが最深の病理)
CLAUDE.md にチェックリスト書いた直後にルール破る。「5 項目チェック」を形式的に埋めて中身ゼロ。  
**対策**: 中身を埋める。形式的だったら自己停止。**実行結果コピペで証明** (CLAUDE.md R3)。

### F8. 全機能を一度に再設計しようとする
v2→v3 で 12 タブ全部作り直そうとして Home/Sessions しか完成せず。v4 で同じパターンを繰り返しかけた。  
**対策**: ROADMAP §0.3 段階実装、各 Stage で「使える状態」を保つ。

### F9. 設計書を読まず実装する (R3 違反)
実装を始めてから「これはどう扱うんだっけ」で設計書を探す。結果、設計書と実装が乖離。

### F10. 中断を軽率に提案する
ユーザーが「先延ばし嫌い」と明言しているのに「次のセッションで」「今日は中断」と提案。  
**対策**: 中断を提案せず完結させる。どうしても必要なら理由明示。

## S11 で発覚した新パターン (2026-04-25)

### F11. agent 報告の鵜呑み (R3 違反)
3 体の Explore agent に v2/v3 リサーチを並列依頼し、報告をそのまま採用。  
- 結果: F1.4.1 (ゲーム単位スコア+状況メモ) が「v2/v3 共に未実装」と誤判定 (実際は v2 完全実装、v3 で削除)
- 根本: agent はファイル全体を読まず該当 keyword 周辺だけ見ることがある  
**対策**: agent 報告の主要結論は **自分で grep / Read で裏取り**。特に「無い」「未実装」の主張は誤判定リスク高。

### F12. 既存ファイルの中身を読まずに「無い」と判断 (R3 違反)
ファイル一覧 (`ls`) で `domain/` に 1 ファイルしか無いことから「v4 に Firestore helper は無い」と推測し、`src/domain/firestore_clean.js` を新規作成。  
- 結果: `src/core/03_storage.js` に既存の `cleanForFirestore` と重複定義、build 時 SyntaxError
- 根本: ファイル名だけで「無い」と判断、中身を読まなかった  
**対策**: 新規追加識別子は **必ず src/ 全体を grep**。CLAUDE.md R3 に明記。

### F13. ユーザー視点の欠落 (R4 違反)
機材セクションのフィールドを「ラケット | テンション縦 / 縦糸 | 横糸 / テンション横 (ぶら下がり)」と機械的に並べた。  
- 問題: ラケット名は長く 1 行欲しい / テンション縦と横は 1 セットなので隣接すべき / 縦糸と横糸も 1 セット  
- 修正: 「ラケット 1 行 / 縦糸+横糸 / テンション縦+横」に統一  
**対策**: フィールド配置時、ラベル名 1 個ずつではなく **「ユーザーが何をどう書きたいか」のグループ単位で考える**。

### F14. 責任を周辺に分散する応答パターン (R5 違反)
失敗の説明で「agent 報告」「HANDOFF」「文書量」「環境」のような外部要因を並べ、「私個人の問題ではない」と読める応答に整理してしまう。  
- 例: 前セッションで「pre-commit hook で物理的に防ぐ」と提案したが、これは「私の自主性に依存しない」=「私を信じないでくれ」という責任放棄構造だった  
**対策**: CLAUDE.md R5。応答主語を「私が」で固定。外部要因を挙げる時は「それを鵜呑みにした私が悪い」とセット。
