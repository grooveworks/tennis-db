# Tennis DB v4.0 — 実装ロードマップ

最終更新: 2026-04-21（Sessions タブ再スコープ） / 前提: REQUIREMENTS_v4.md, ARCHITECTURE_v4.md, CLAUDE.md

## 0. 基本方針（v3 の失敗を踏まえた転換）

### 0.1 UI/UX 先行
**デザインシステムとワイヤフレームが確定するまで、機能実装のStageに入らない**。
- v3 は機能実装を先行したため、デザインが後回しになり、最後まで UI 統一感が出なかった
- v4 はこの順序を逆転する
- デザイン確定前に機能を作りたい欲求が出たら、ストップサイン発動

### 0.2 v3 Firestore 共有
**v3 と v4 は同じ Firestore collection を使う**（データ移行不要）。
- `users/{uid}/data/{key}` の構造は v3 と同一
- ユーザーは URL 切替だけで v3/v4 を行き来できる
- v4 でバグったら v3 に戻れる安全網
- 最終的に v3 を凍結、v4 に完全移行するが、それまで並行運用

### 0.3 段階実装（各 Stage で使える状態を保つ）
- S5 以降は「各 Stage 終了時点で v4 が何らかの役に立つ」状態を維持
- 全機能完成を待たず、段階的に v3 → v4 へ使用を移していける

### 0.4 対処療法の即時停止
- 規律違反（CLAUDE.md §「厳守プロトコル」参照）を感知した瞬間、作業を止めて報告
- 「とりあえず」「後で直す」は禁止
- 詳細は CLAUDE.md と ARCHITECTURE_v4.md §10 参照

### 2026-04-21 の再スコープ記録

当初は S5「Sessions 一覧（読むだけ）」の 1 Stage で UI まで完結させる想定だった。着手中に 950 件の実データを前にして以下が浮上:

- フラットリスト＋月グループだけでは 3 年分の活動は探索できない
- 検索・絞り込み・視覚化（カレンダー・年間濃淡）が無いと v3 の延長にしかならない
- v3 Sessions タブの問題（情報詰め込み・視覚ノイズ・展開でスクロール崩壊・結果の重み付け平坦）を引き継ぐと **v4 を作る意味が無い**
- カードタップの展開を「accordion」から「画面遷移」に変更（リスト崩れない、戻り時にスクロール位置復元）

結果、旧 S5 を S5-S10 の 6 Stage に分解。元 S7 以降を S11 以降にシフト。S20 に自動連関を新規追加。

### 2026-04-21 の日付統一方針（v4 全体に影響）

- v4 の canonical 日付形式は **`YYYY-MM-DD`**（ISO 8601、HTML date 入力／Google カレンダー／CSV の標準と同じ）
- v4 が v3 の古い `YYYY/M/D` データに触れた時、その場で `YYYY-MM-DD` に書き直して保存する（遅延マイグレーション）
- v3 は元から両形式を読める（normDate が adapter）ので、v4 が先に新形式で書いても v3 で壊れない
- S21（v3 凍結時）に残った古い形式データを一括書き換え

### 2026-04-21 の画面幅対応方針（Sessions タブ以降に影響）

- **幅 < 600px（スマホ）**: 単列、親指優先、スライドイン詳細＋戻り経路 4 つ（スワイプ・下バー・タブ再タップ・Esc）
- **600〜1023px（小型タブレット縦・スマホ横）**: スマホ挙動のまま、カードの情報密度を少し上げる
- **幅 ≥ 1024px（PC・タブレット横）**: 左右分割、「戻る」行為そのものを廃止

v2/v3 で先送りされていた「戻るが遠い問題」を画面幅に応じた表示切替で根本解決する。ブラウザ戻るボタンに依存しない。

### 2026-04-21 の S11（Session 編集画面）前提要件

使用状況の整理（ユーザー提示）:
- PC（家・仕事場）: 詳細登録、Google カレンダー同期、データ保守
- スマホ（出先）: ホーム 3 ボタンで簡易登録、練習後の詳細登録
- ミニタブレット（試合中）: ゲーム単位スコアと状況メモの記録（ポイント単位は DataTennis で別途記録、後で CSV 取込）

これに伴い S11 の前提要件:
- ゲーム単位の試合記録 UX（REQUIREMENTS F1.4.1 参照）
- ゲーム間の 30 秒で入力可能な最小タップ導線
- 屋外強光下でも読めるコントラスト、誤タップしにくい大ボタン
- 書いた直後に読み返しやすい配置（メンタルリセット目的）
- S11 着手前に個別 HANDOFF で UX を先に設計する（UI/UX 先行原則）

関連 Stage:
- S19（インポート）: DataTennis CSV の取込と、既存のゲーム単位記録との統合

## 2. 各 Stage の完了条件（DoD）

### 共通 DoD（全 Stage）
1. 実装完了後、preview で動作確認（S2-S3 は文書確認）
2. 受入基準6項目を満たす（REQUIREMENTS_v4.md §5）
3. ユーザー検証OK（手順は私が用意、ユーザーは「OK/NG」判定のみ）
4. 1 commit にまとめる
5. push 承認 → push
6. 次 HANDOFF 作成 + 現 HANDOFF 削除

### Stage 個別 DoD

#### S2: デザインシステム定義
- `DESIGN_SYSTEM_v4.md` 作成、以下を明文化:
  - 色パレット（主/副/補助の3段階、AA 以上、主要は AAA）
  - タイポグラフィ（見出し/本文/補助、フォントサイズ階層、行間、文字間）
  - 余白（基本単位、コンポーネント内/間の余白）
  - コンポーネント仕様（ボタン/カード/入力欄/モーダル/タブバー の形・サイズ・状態変化）
  - アイコンの扱い方
  - モーション（transition 時間・イージング）
  - タップ領域（最小 44×44px）
- ユニバーサルデザイン要件（色弱・文字サイズ可変・キーボード操作・スクリーンリーダー）

#### S3: ワイヤフレーム
- `WIREFRAMES_v4.md` 作成（Markdown+ASCII アート or 画像リンク）:
  - 全画面のレイアウト案（Home / Sessions一覧 / Session展開 / Detail / 各モーダル）
  - 戻るボタン・主アクション・メニューの固定位置
  - 画面遷移図（どこからどこへどう行けるか）

#### S4: 共通UIコンポーネント
- `src/ui/common/` に Badge/Card/Button/Input/Select/Modal/Toast/ConfirmDialog 実装
- DESIGN_SYSTEM_v4.md に完全準拠
- tests/run.html で単体動作確認

#### S5-S16 の DoD は各 HANDOFF で個別定義
着手直前の HANDOFF で、その Stage の具体的 DoD と検証手順を記載。

## 3. 各 Stage 着手前の必須手順（守らないと作業しない）

以下を応答テキストに明示的に書き出してから実装を開始する。詳細は CLAUDE.md §「厳守プロトコル」。

```
□ 関連設計書を読んだ(読んだ要点3行)
□ 実行環境の制約を調査した(preview/file://, PowerShell, Firebase 等)
□ DoD の各項目が preview で検証可能か判定(○×で)
□ 使用技術の仕様を調査した(具体的な罠を書き出す)
□ ユーザー承認を得た
```

書き出した内容が形式的・空ならその時点で自己停止 → ユーザー報告。

## 4. v3 → v4 の並行運用とリリース判断

### 並行運用期間（S5 〜 S15）
- v3: `https://grooveworks.github.io/tennis-db/v3/` 本番継続
- v4: `https://grooveworks.github.io/tennis-db/v4/` 開発版
- 両方とも同じ Firestore を読む → データ一貫性保持
- v4 でバグ発覚時は URL を v3 に戻すだけ（5秒復旧）

### v4 への切替判断（S12 完了時点）
- S12 完了時点で「v4 だけで記録 → 表示 → 編集 → 追加」が完結
- この時点でユーザーが「v4 常用」に切り替え可能
- v3 は「万一用」として残す

### v3 凍結（S21）
- S16-S18 完了で v3 を超える機能揃う
- S21 で v3 を正式凍結（index.html の v3/ ディレクトリは残すが新機能追加しない）
- 並行運用期間で書き換わらなかった古い日付形式データを一括マイグレーション

## 5. 作業規律（対処療法を繰り返さないために）

CLAUDE.md §「厳守プロトコル」に詳細記載。要約:

1. 設計書を読み直してから実装開始（毎 Stage）
2. type 分岐は domain 層の1箇所のみ、UI で書いたら却下
3. エラーは根本原因を特定してから直す
4. 「とりあえず」「後で直す」禁止
5. 判断を投げない（「こうします」と宣言）
6. 先延ばし禁止（中断を軽率に提案しない）
7. 同一ファイル 3 回目の Edit をしようとしたら一度止まる
8. テンプレ的応答禁止（状況に応じた判断）

## 6. 失敗時のエスカレーション

- Stage 内で設計見直しが 3 回以上発生 → REQUIREMENTS/ARCHITECTURE に立ち返って前提を疑う
- 規律違反を自覚 → 即時ユーザー報告、作業中断
- 致命的データロスリスク → 即停止、ユーザー判断仰ぐ

## 7. タイムライン（見込み）

1 Stage = 1-2 セッション想定。22 Stage で **25〜30 セッション**程度。

- S2-S5（デザイン先行）は実装前の重要投資で、時間をかけてよい
