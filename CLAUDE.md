# Tennis DB プロジェクト — Claude 必須ルール（厳守プロトコル）

**このファイルは毎セッション開始時に私（Claude）が最初に読む**。
**規律違反を自覚した瞬間、即時停止してユーザーに報告する**。

---

## 0. 最優先ルール（他の全てに優先）

### 0.1 対処療法禁止
問題が発生したら、その瞬間の症状を直すのではなく、**根本原因を調査してから直す**。
「とりあえず動く」「後で直す」を作らない。

### 0.2 判断を投げない
「A/B/C どちらにしますか?」を聞いてはいけない。
**「こうします。違えば指示ください」**の形で提示する。
素人ユーザーに技術判断を求めるのは傲慢。

### 0.3 事前調査してから実装
実行環境（preview/file://）、使用技術（PowerShell の罠、Firebase の制約等）を調査してから設計・実装。
調査なしに書き始めない。

### 0.4 UI/UX 先行
デザインシステム (ROADMAP S2-S4) が完成するまで、機能実装 Stage に進まない。
「機能を作ってから見た目」は禁止。

### 0.5 謝罪は単独で
失敗時、「すみません、次は○○します」のように次アクションと混ぜない。
謝罪は単体、その後に改めて次の行動を提示。

### 0.6 専門用語でユーザーに説明しない
ユーザーは素人。専門用語（Stage, Domain, カバレッジ, Firestore 等）を使わず平易に書く。
設計書は私の作業用、ユーザーへの説明は別文書で。

## 1. コード変更前の強制チェックリスト（実装着手前、必須）

Edit/Write でコードに触る前、以下を応答テキストに**中身を埋めて**書き出す。
形式的に埋めたらルール違反＝自己停止。

```
□ 関連設計書を読んだ
  → REQUIREMENTS_v4.md / ARCHITECTURE_v4.md / ROADMAP_v4.md の該当部分の要点3行

□ 実行環境の制約を調査した
  → preview/file://, PowerShell, Firebase 等、該当する制約を具体列挙

□ DoD の各項目が preview で検証可能か判定
  → 各 DoD 項目を ○×で判定、× なら別の検証方法を明示

□ 使用技術の仕様を調査した
  → 具体的な罠（例: PowerShell の -join は環境依存、Firebase は file:// で動かない）

□ ユーザー承認を得た
  → どの指示で着手許可を得たか明示
```

## 2. ストップサイン（対処療法の兆候、発動したら即停止）

以下のどれかを自覚したら、**その瞬間に手を止めてユーザー報告**。

| # | シグナル | 対応 |
|---|---|---|
| 2.1 | 同一ファイルに 3 回目の Edit | 「このファイルに 3 回目の編集です。根本から見直すべきかもしれません」と報告 |
| 2.2 | エラーに対して「これで回避できる」と思った | 回避策を採らず、根本原因調査に戻る |
| 2.3 | 「とりあえず」「後で直す」「一旦」と書きそうになった | 書かない、ちゃんとした解決策を考える |
| 2.4 | 判断をユーザーに投げそうになった | 自分の判断案を出してから「違えば指示ください」の形に書き換え |
| 2.5 | 事前調査なしに実装開始しそうになった | チェックリスト（§1）を埋めてから再開 |
| 2.6 | UI/UX を機能実装より後回しにしそう | デザインシステム（S2-S4）未完なら機能実装に進まない |
| 2.7 | 設計書にない機能を実装しそう | REQUIREMENTS に追記してユーザー承認を得てから |
| 2.8 | 環境で動かないことを DoD に含めた | DoD を検証可能な形に書き直す |

## 3. コミット・push 規律

### 3.1 コミット粒度
- 1 タスク = 1 push が原則
- 小刻み patch 禁止
- 関連バグは先回りで洗い出して同一 commit
- 「push→動作確認→バグ→小 patch→push」サイクルを作らない

### 3.2 push 前確認
- push 前にユーザーから**明示的承認**を必ず取る
- 「OK」「やってください」が何のスコープに対してかを再確認
- バンドルした承認禁止（1 コミット = 1 承認）

### 3.3 push 後
- push できたら HANDOFF を更新（次セッション用）

## 4. 繰り返されてきた失敗パターン（再発防止のため記録）

### 4.1 対処療法の連鎖（v2→v3→v4 で繰り返し）
- 問題発覚 → パッチ → 新たな歪み → パッチ、の無限ループ
- 結果、v3 は Sessions タブの細部だけ発達し、Gear/Plan/Insights は Placeholder のまま
- **対策**: §2 のストップサイン、§0.1 の最優先ルール

### 4.2 判断をユーザーに投げる
- 「A/B/C どれ?」を連発して素人に技術判断を求める
- 指摘されて「判断します」と誓った後、また投げる
- **対策**: §0.2 の最優先ルール、§2.4 のストップサイン

### 4.3 事前調査不足で動かない DoD を書く
- 環境制約（preview/file://）を調べず、Firebase 動作を DoD に入れる
- PowerShell の `-join` / `Set-Content` の罠に時間浪費
- **対策**: §0.3 の最優先ルール、§1 のチェックリスト §2.5, §2.8

### 4.4 謝罪を軽く流す
- 「ミスでした、でも○○なら OK です」と言い訳にすり替える
- 謝罪と次アクションを同じ応答で混ぜる
- **対策**: §0.5 の最優先ルール

### 4.5 専門用語で素人にを置き去りにする
- 「Stage」「Domain」「カバレッジ」等の用語を平易化せずに使う
- ユーザーが理解していないのに承認を求める
- **対策**: §0.6 の最優先ルール

### 4.6 デザインを軽視する
- 機能優先で作り、デザインは後回し
- v3 ではデザインに手が回らず、UI 統一感なく終わった
- **対策**: §0.4 の最優先ルール、§2.6 のストップサイン

### 4.7 文書を書いて満足する
- CLAUDE.md にチェックリスト書いた直後にルール破る
- 「5項目チェック」を形式的に埋めて中身ゼロ
- **対策**: 中身を埋める、形式的だったら §1 違反として自己停止

### 4.8 全機能を一度に再設計しようとする
- v2→v3 で 12 タブ全部作り直そうとして Home/Sessions しか完成せず
- v4 で同じパターンを繰り返しかけた
- **対策**: ROADMAP §0.3 段階実装、各 Stage で「使える状態」を保つ

### 4.9 設計書を読まず実装する
- 実装を始めてから「これはどう扱うんだっけ」で設計書を探す
- 結果、設計書と実装が乖離
- **対策**: §1 チェックリストの1項目目

### 4.10 中断を軽率に提案する
- ユーザーが「先延ばし嫌い」と明言しているのに「次のセッションで」「今日は中断」と提案
- **対策**: 中断を提案せず完結させる、どうしても中断が必要なら理由明示
