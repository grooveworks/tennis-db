# Tennis DB v4.0 — 要件定義書

作成日: 2026-04-19 / バージョン目標: v4.0.0

---

## 0. この文書の目的

v3 が対処療法の積み重ねになった反省から、**コードを書く前に「何を作るか」を確定させる**ための文書。ここに書かれていない機能は実装しない。書いた機能は必ず受入基準まで到達させる。

---

## 1. アプリの目的

個人テニスプレイヤーの日常を **記録 → 振り返り → 次の意思決定** に変換する支援ツール。埼玉県在住競技志向プレイヤー（INTJ × 高O × 高H）が本業の合間に使う前提。

**最優先価値**: タイピング・操作回数の最小化。スピードより確実性。データロス0。

---

## 2. 想定ユーザー

- プレイヤー本人（唯一のユーザー）
- スポーツカウンセラーへの資料提供先（読み取り専用の将来用途）

## 3. 機能要件

### F1. セッション記録（コア機能）
- **F1.1** 大会 (tournament) の作成・編集・削除
- **F1.2** 練習 (practice) の作成・編集・削除
- **F1.3** 試打 (trial) の作成・編集・削除
- **F1.4** 大会配下の試合 (match) の個別追加・編集・削除
- **F1.5** 試打と練習・試合の連携（linkedPracticeId / linkedMatchId）
- **F1.6** セッションの一覧表示（直近3ヶ月デフォルト + 年月ジャンプ）
- **F1.7** セッションの展開表示（タップで詳細概要、再タップで閉じる）
- **F1.8** セッションの全画面編集モード（Detail ページ）
- **F1.9** セッションの一括削除（編集モードでチェックボックス選択）
- **F1.10** セッションのマージ（同タイプ2件を1件に統合、cascade 削除）

### F2. 機材管理
- **F2.1** ラケット管理（所有履歴、セッティング記録、使用頻度）
- **F2.2** ストリング管理（ゲージ、素材、推奨テンション）
- **F2.3** セッティング組合せ管理（ラケット × ストリング × テンション）
- **F2.4** 実測値ログ（ガット張替時の実測、フレーム重量実測）

### F3. 計画管理
- **F3.1** Next Actions（短期TODO、優先度付き）
- **F3.2** 対戦相手管理（名前、特徴、戦績）
- **F3.3** 試合・練習計画（将来）

### F4. ホーム
- **F4.1** 3つの大きな登録ボタン（大会/練習/試打）
- **F4.2** Next Actions 上位表示
- **F4.3** MiniCalendar（近日予定）
- **F4.4** 最近の好成績
- **F4.5** バックアップ警告（エクスポートから日数経過で表示）

### F5. 分析・振り返り
- **F5.1** スタッツ集計（勝率、種別別、期間別）
- **F5.2** メンタル・フィジカルの推移
- **F5.3** ラケット・ストリング別のパフォーマンス
- **F5.4** プロフィール（利き手、目標クラス）

### F6. データ入出力
- **F6.1** エクスポート（全データJSON、タイムスタンプ付き）
- **F6.2** インポート（JSON、バックアップ復元）
- **F6.3** GoogleカレンダーJSONインポート（予定の一括取込）
- **F6.4** データテニスCSVインポート（試合スタッツ）
- **F6.5** Apple Watch ワークアウトデータインポート（心拍ゾーン）
- **F6.6** インポートプレビュー（新規/重複可視化、per-item 承認）
- **F6.7** インポート取り消し（直前操作のロールバック）

### F7. 同期・認証
- **F7.1** ログイン不要でlocalStorageのみで動作
- **F7.2** Googleログイン後、Firestore と同期（クラウドバックアップ）
- **F7.3** 複数端末間で同期

### F8. Claude 連携（将来）
- **F8.1** 記録内容を Claude に渡す形式で整形・コピー
- **F8.2** カウンセリング向けレポート出力

## 4. 非機能要件

### N1. ユーザー体験
- **N1.1** iPhone 片手親指操作を前提、タップ領域最小44px
- **N1.2** モーダル開閉で state 喪失しない（スクロール位置・入力途中・選択状態の保持）
- **N1.3** 連続タップ・高速操作で state 不整合が起きない
- **N1.4** 視認性: **白/淡色背景上の細字は textMuted 禁止**（AAA 7:1 以上確保）
- **N1.5** 情報階層は「主 (text) → 副 (textSecondary) → 補助 (textMuted)」の3段で統一

### N2. データ信頼性
- **N2.1** 本番データロス0が最優先。破壊的操作は確認ダイアログ必須
- **N2.2** Session削除時、参照する trial.linkedXxx を自動クリア（孤児化禁止）
- **N2.3** match削除時、参照する trial.linkedMatchId を自動クリア
- **N2.4** Import commit は per-item 即時、モーダル閉じても処理済みは保存済み
- **N2.5** 500件規模のデータで UI が破綻しない
- **N2.6** Firestore 書き込み失敗時はユーザーに通知

### N3. 保守性
- **N3.1** 単一ファイル3000行超を禁止。論理単位でファイル分割
- **N3.2** データスキーマは単一の真実（SCHEMA）、新フィールドは1行追加で完結
- **N3.3** マージ・削除・比較ロジックは中央ヘルパー化、type 分岐を散らさない
- **N3.4** domain ロジックは副作用なし（純関数）、React に依存しない
- **N3.5** 削除・マージは domain 層の unit test を必ず書く

### N4. デプロイ・運用
- **N4.1** GitHub Pages で配信可能（ビルド後の成果物は単一 index.html が理想）
- **N4.2** ビルドコマンドは PowerShell 1行で完結（Node 依存最小）
- **N4.3** ローカル serve は serve.ps1 で 1 コマンド
- **N4.4** ロールバックは git revert + 再push で完結

## 5. 受入基準（各機能共通）

各機能は以下を全て満たすまで「完成」とみなさない:

1. ✅ **単体動作**: 機能単独で期待通りに動く
2. ✅ **連続操作**: 開閉・キャンセル・再開を3回繰り返しても破綻しない
3. ✅ **大量データ**: 500件規模で UI の応答時間が1秒以内
4. ✅ **異常系**: 空データ・欠損値・不正値でエラーにならない
5. ✅ **データ整合性**: 関連データ（linkedXxx 等）が cascade で正しく更新される
6. ✅ **domain 層の test**: 純関数部分は unit test で網羅
7. ✅ **ユーザー検証**: preview で最低3シナリオ実地確認、ユーザーが「OK」と返答

6 の unit test は domain 層（merge / cascade / import / schema 検証）に限定。UI は手動検証。

## 6. スコープ外（v4.0 では実装しない）

- 双方向同期（GCal ↔ Tennis DB）
- Make.com Webhook 連携
- 戦績・成長ログの自動生成
- スポーツカウンセラー向けレポート自動生成
- PWA 化・通知

これらは v4.1 以降のバックログに置く。

---

## 7. v3 から引き継ぐデータ

既存 Firestore の本番データ（大会/練習/試打/機材/Next/対戦相手/プロフィール/会場一覧）は v4.0 形式に移行する。移行スクリプトは v4 実装の最終段階で作成。移行時のデータロス0が絶対条件。

---

## 8. この文書の変更ルール

- 機能追加・削除は必ずこの文書を更新してから実装
- 受入基準を満たす前に「完成」と宣言しない
- 文書と実装の乖離を発見したら、実装ではなく文書を正とする（仕様起点）
