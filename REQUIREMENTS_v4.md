# Tennis DB v4.0 — 要件定義書

作成日: 2026-04-19 / 最終更新: 2026-04-19 / バージョン目標: v4.0.0

---

## 0. この文書の目的

v3 が対処療法の積み重ねになった反省から、**コードを書く前に「何を作るか」を確定させる**ための文書。ここに書かれていない機能は実装しない。書いた機能は必ず受入基準まで到達させる。

## 0.1 v4 を作る意味（v3→v4 の革新ポイント）

v2→v3 は「12タブ→5タブ統合」という明確な UX 革新があった。
v3→v4 も同等以上の革新が必要。単なるコード品質改善では v4 を作る意味が薄い。

### v4 の価値提案（ユーザーから見た「何が新しくなるか」）
1. **UI/UX 統一感**: デザインシステム先行で、画面ごとのバラツキを根絶
2. **ユニバーサルデザイン準拠**: 色弱・文字拡大・片手操作すべて配慮
3. **v3 未実装タブの実現**: Gear / Plan / Insights を初めて使える形で提供
4. **データ一貫性**: 削除時の cascade、マージの堅牢化、取り消し可能性
5. **コード品質**: SCHEMA 中央化、分割・テスト可能な構造

1〜3 がユーザー視点の革新、4〜5 が開発者視点の革新。**両方揃えて初めて v4 を作る価値が出る**。

## 1. アプリの目的

個人テニスプレイヤーの日常を **記録 → 振り返り → 次の意思決定** に変換する支援ツール。埼玉県在住競技志向プレイヤー（INTJ × 高O × 高H）が本業の合間に使う前提。

**最優先価値**: タイピング・操作回数の最小化。スピードより確実性。データロス0。

## 3. 機能要件

### F1. セッション記録（コア機能）
- **F1.1** 大会 (tournament) の作成・編集・削除
- **F1.2** 練習 (practice) の作成・編集・削除
- **F1.3** 試打 (trial) の作成・編集・削除
- **F1.4** 大会配下の試合 (match) の個別追加・編集・削除
- **F1.4.1** **試合中のゲーム単位記録**（2026-04-21 追加）
  - 用途: ゲーム間（30 秒程度）に素早くスコアと簡単な状況メモを書く
  - 目的 1: ポイント単位の詳細スタッツは DataTennis アプリが担当（F6.4 で後から CSV 取込）、TennisDB はその保険としてゲーム単位で残す
  - 目的 2: 「試合を客観的に見る」「負けている時のメンタルリセット」の補助
  - 記録項目: 各ゲームのスコア（例: 3-1 など）、そのゲームの状況の短いメモ（例: 「サーブ調子上向き」「集中切れた」「ネット寄ろう」）
  - UX 要求: 最小タップで入力可能、誤タップしにくい大きなボタン、強い日射下でも読みやすい配色、書いた直後に読み返しやすい配置
- **F1.5** 試打と練習・試合の連携（linkedPracticeId / linkedMatchId）
- **F1.6** セッションの一覧表示（3 モード: リスト・カレンダー・年間濃淡。詳細は WIREFRAMES §2.2 参照）
- **F1.7** セッション詳細表示（カードタップで画面幅に応じた表示切替: スマホはスライドイン＋複数戻り経路、PC/タブレット横は左右分割）
- **F1.8** セッションの全画面編集モード（Detail ページ）
- **F1.9** セッションの一括削除（編集モードでチェックボックス選択）
- **F1.10** セッションのマージ（同タイプ2件を1件に統合、cascade 削除）
- **F1.11** セッションの検索・複数軸絞り込み（テキスト検索、ラケット・対戦相手・結果等での複合絞り込み）

### F2. 機材管理
- **F2.1** ラケット管理（所有履歴、セッティング記録、使用頻度）
- **F2.2** ストリング管理（ゲージ、素材、推奨テンション）
- **F2.3** セッティング組合せ管理（ラケット × ストリング × テンション）
- **F2.4** 実測値ログ（ガット張替時の実測、フレーム重量実測）

### F3. 計画管理
- **F3.1** Next Actions（短期TODO、優先度付き）
- **F3.2** 対戦相手管理（名前、特徴、戦績）
- **F3.3** 試合・練習計画（将来）

### F4. ホーム
- **F4.1** 3つの大きな登録ボタン（大会/練習/試打）
- **F4.2** Next Actions 上位表示
- **F4.3** MiniCalendar（近日予定）
- **F4.4** 最近の好成績
- **F4.5** バックアップ警告（エクスポートから日数経過で表示）

### F5. 分析・振り返り
- **F5.1** スタッツ集計（勝率、種別別、期間別）
- **F5.2** メンタル・フィジカルの推移
- **F5.3** ラケット・ストリング別のパフォーマンス
- **F5.4** プロフィール（利き手、目標クラス）

### F6. データ入出力
- **F6.1** エクスポート（全データJSON、タイムスタンプ付き）
- **F6.2** インポート（JSON、バックアップ復元）
- **F6.3** GoogleカレンダーJSONインポート（予定の一括取込）
- **F6.4** データテニスCSVインポート（試合スタッツ）
- **F6.5** Apple Watch ワークアウトデータインポート（心拍ゾーン）
- **F6.6** インポートプレビュー（新規/重複可視化、per-item 承認）
- **F6.7** インポート取り消し（直前操作のロールバック）

### F7. 同期・認証
- **F7.1** ログイン不要でlocalStorageのみで動作
- **F7.2** Googleログイン後、Firestore と同期（クラウドバックアップ）

## 4. 非機能要件

### N1. ユーザー体験 / UI / UX（v3 の最大の反省点を反映）
- **N1.1** iPhone 片手親指操作を前提、タップ領域最小44×44px
- **N1.2** モーダル開閉で state 喪失しない（スクロール位置・入力途中・選択状態の保持）
- **N1.3** 連続タップ・高速操作で state 不整合が起きない
- **N1.4** 視認性: **白/淡色背景上の細字は textMuted 禁止**（AAA 7:1 以上確保）
- **N1.5** 情報階層は「主 (text) → 副 (textSecondary) → 補助 (textMuted)」の3段で統一
- **N1.6** **デザインシステム先行**: 機能実装より先にデザインシステム確定（ROADMAP S2）
- **N1.7** **画面統一性**: 全画面で戻る・主アクション・メニューの位置固定、コンポーネント形状統一
- **N1.8** **暗い・モノトーンは避ける**: 明るい基調、アクセントで情報重要度を表現
- **N1.9** **情報垂れ流しを避ける**: LINE/SNS 的な「時系列フラット」は作らない。階層・まとまりを設計
- **N1.10** **画面全体のバランス**: 情報量に応じた文字間・余白の調整、密集回避
- **N1.11** **アニメーションは適度に**: 状態変化のフィードバックとして使う、派手すぎず無さすぎず
- **N1.12** **無駄な確認ダイアログ禁止**: 破壊的操作のみ確認、それ以外は即時実行

### N1a. ユニバーサルデザイン要件
- **N1a.1** 全色ペア WCAG AA 以上、小さい文字 (12px以下) は AAA 必須
- **N1a.2** 色だけで状態を示さない（必ずアイコン or 文字ラベル併用）
- **N1a.3** 文字サイズ拡大（ブラウザ設定 200% 等）に対応
- **N1a.4** キーボード操作で全機能アクセス可能（Tab / Enter / Esc）
- **N1a.5** スクリーンリーダー読み上げで操作フロー理解可能（aria 属性）
- **N1a.6** タップ領域最小 44×44px（iOS HIG 準拠）
- **N1a.7** 動きに弱いユーザー向け `prefers-reduced-motion` 対応

### N2. データ信頼性
- **N2.1** 本番データロス0が最優先。破壊的操作は確認ダイアログ必須
- **N2.2** Session削除時、参照する trial.linkedXxx を自動クリア（孤児化禁止）
- **N2.3** match削除時、参照する trial.linkedMatchId を自動クリア
- **N2.4** Import commit は per-item 即時、モーダル閉じても処理済みは保存済み
- **N2.5** 500件規模のデータで UI が破綻しない
- **N2.6** Firestore 書き込み失敗時はユーザーに通知

### N3. 保守性
- **N3.1** 単一ファイル3000行超を禁止。論理単位でファイル分割
- **N3.2** データスキーマは単一の真実（SCHEMA）、新フィールドは1行追加で完結
- **N3.3** マージ・削除・比較ロジックは中央ヘルパー化、type 分岐を散らさない
- **N3.4** domain ロジックは副作用なし（純関数）、React に依存しない
- **N3.5** 削除・マージは domain 層の unit test を必ず書く

### N4. デプロイ・運用
- **N4.1** GitHub Pages で配信可能（ビルド後の成果物は単一 index.html が理想）
- **N4.2** ビルドコマンドは PowerShell 1行で完結（Node 依存最小）
- **N4.3** ローカル serve は serve.ps1 で 1 コマンド
- **N4.4** ロールバックは git revert + 再push で完結

## 5. 受入基準（各機能共通）

各機能は以下を全て満たすまで「完成」とみなさない:

1. ✅ **単体動作**: 機能単独で期待通りに動く
2. ✅ **連続操作**: 開閉・キャンセル・再開を3回繰り返しても破綻しない
3. ✅ **大量データ**: 500件規模で UI の応答時間が1秒以内
4. ✅ **異常系**: 空データ・欠損値・不正値でエラーにならない
5. ✅ **データ整合性**: 関連データ（linkedXxx 等）が cascade で正しく更新される
6. ✅ **domain 層の test**: 純関数部分は unit test で網羅
7. ✅ **ユーザー検証**: preview で最低3シナリオ実地確認、ユーザーが「OK」と返答

6 の unit test は domain 層（merge / cascade / import / schema 検証）に限定。UI は手動検証。

## 7. v3 との関係（データ・運用）

### 7.1 Firestore 共有（データ移行なし）
- v3 と v4 は**同じ Firestore collection を共有**する（`users/{uid}/data/{key}`）
- データ構造は v3 と互換、SCHEMA 拡張で新フィールド追加も可（SCHEMA に追加するだけで既存データとも共存）
- **移行スクリプト不要**。ユーザーは URL を切り替えるだけで同じデータを v3/v4 で見られる

### 7.2 並行運用
- v3: `https://grooveworks.github.io/tennis-db/v3/` 本番継続（凍結はしない、バグ修正のみ）
- v4: `https://grooveworks.github.io/tennis-db/v4/` 開発版
- S8 完了で v4 実運用可能（記録/閲覧/編集/追加/削除）
- S12-14 完了で v3 を超える機能揃う
- S16 で v3 を正式凍結

### 7.3 v4 に v3 未達機能を初実装
v3 では Gear / Plan / Insights タブが Placeholder（件数表示のみ）のまま未実装だった。
v4 で初めてこれらを実装する（S12-S14）。v3 の実装に引きずられず、クリーンな設計で作れる。
